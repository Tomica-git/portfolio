# 半導体メモリ検査工程の業務効率化 ― Excel 関数 + VBA による自動化

> 技術派遣先での破断面観察業務において、画像ファイル名の命名規則設計と  
> Excel 関数・VBA を組み合わせた自動抽出の仕組みを構築し、  
> **手作業 3〜4 時間 → 約 45 分**へ短縮した取り組みの記録です。

---

## 背景

半導体メモリの品質管理工程のひとつに**破断面観察**があります。  
サンプルの断面をマイクロスコープで撮影し、取得した画像データを約 1,000 行ある Excel 管理台帳に記録、最終的に TIBCO Spotfire（DXP）でデータを集計・可視化するという一連のフローです。

### 従来の課題

- 撮影 → 保存 → Excel 手入力 → Spotfire 集計 のすべてが手作業
- 1,000 行超のシートから該当セルを**目視で検索**し、データを 1 件ずつ手入力
- 転記ミスが頻発し、ミス修正に **3〜4 時間**かかることがあった
- Excel の入力ミスが Spotfire の集計にも波及し、やり直しが発生

---

## 解決アプローチ

### 1. ファイル命名規則の設計

マイクロスコープで画像を保存する際に、後工程で必要なデータを**略語で埋め込む**命名規則を自ら設計しました。

```
例: 84-50TE04①_X.XXum破断状態A.jpg
```

| 位置 | 内容 | 抽出例 | 意味 |
|------|------|--------|------|
| 1〜5 文字目 | ウェハ型番（略称） | `84-50` | → 正式型番 `DCH003484-50` へ変換 |
| 6〜7 文字目 | ウェハ上の取得位置 | `TE` | → Top Edge（上・外側）等 |
| 8〜9 文字目 | サンプル番号 | `04` | → 4 番目のサンプル |
| 10 文字目 | 破断方向 | （3 パターン） | → IFERROR + FIND で判定 |
| `_` 〜 `um` の間 | 測定値 | `X.XX` | → FIND + MID + ×1 で数値変換 |
| `um` 〜 `.jpg` の間 | 破断状態 | `破断状態A` | → 破断起点のフィルター用分類 |

> ポイント: ファイル名自体を**構造化データのソース**にすることで、手入力を排除する設計です。

---

### 2. マクロ付き Excel（自動入力マクロ.xlsm）による運用フロー

VBA マクロとボタンを組み込んだ `.xlsm` ファイルで、以下の手順で操作します。

#### 運用手順

| 手順 | 操作 | 結果 |
|------|------|------|
| ① | 「操作」シートの**【選択するフォルダを決定】**ボタンを押す | フォルダ選択ダイアログが開く |
| ② | 画像ファイルのあるフォルダを選択 | C5 セルに選択したフォルダのパスが表示される |
| ③ | **【選択したフォルダからfilelistのシートにファイル名を張り出す】**ボタンを押す | 「filelist」シートの A 列に全ファイル名が上から下へ出力される |
| ④ | filelist の A 列を全選択 → コピー | ファイル名をクリップボードにコピー |
| ⑤ | 「貼り付け場所変換」シートの A 列に貼り付け | B 列以降の関数が自動で各フィールドを抽出 |

> 詳細は [`src/自動入力マクロ.xlsm`](src/自動入力マクロ.xlsm) および [`src/自動入力マクロ.bas`](src/自動入力マクロ.bas)（VBA ソースコード）を参照してください。

### 3. Excel 関数によるファイル名パース

取得したファイル名から、Excel 関数を使って各フィールドを自動抽出しました。

#### ウェハ型番（1〜5 文字目 → IF ネストで正式型番へ変換）

```excel
=IF(MID(A2,1,5)="80-50","DH260507580-50",
 IF(MID(A2,1,5)="84-12","CE260402384-12",
  IF(MID(A2,1,5)="84-50","DCH003484-50","該当なし")))
```

`MID` で抽出した 5 文字を `IF` のネストで略称→正式型番へ変換。  
どの条件にも一致しなければ「該当なし」を返す。  
当初は `LET + IFS` で実装したが、旧バージョンの Excel で `@` 演算子が自動挿入され `#NAME?` エラーが発生したため、**全バージョンで動作する IF ネスト**に変更した。

#### 取得位置（6〜7 文字目 → IF ネストで正式名へ変換）

```excel
=IF(MID(A2,6,2)="TE","Top Edge",
 IF(MID(A2,6,2)="BM","Bottom Middle",
  IF(MID(A2,6,2)="Ct","Center",
   IF(MID(A2,6,2)="LE","Left Edge",
    IF(MID(A2,6,2)="RE","Right Edge","該当なし")))))
```

`Edge`（外側）・`Middle`（中央）・`Center`（真ん中）と、Top / Right / Bottom / Left の 4 方向を組み合わせて取得位置を判断。

| コード | 変換結果 |
|--------|---------|
| `TE` | Top Edge（上・外側） |
| `BM` | Bottom Middle（下・中央） |
| `LE` | Left Edge（左・外側） |
| `RE` | Right Edge（右・外側） |
| `Ct` | Center（ど真ん中。範囲が狭いため 4 方向の区分なし） |

#### サンプル番号（8〜9 文字目 → *1 で数値変換）

```excel
=MID(A2, 8, 2) * 1
```

`MID` で抽出した文字列 `"04"` を `* 1` で数値 `4` に変換。Spotfire での集計時に数値型として扱えるようにする。

#### 破断方向（10 文字目）― IF + IFERROR + FIND による堅牢な判定

パターンが 3 種類に限られていたため、単純な `MID` で 1 文字抽出するのではなく、**IFERROR と FIND の組み合わせ**でエラーに強い判定式を設計しました。どのパターンにも該当しない場合は「該当なし」を返します。

```excel
=IF(IFERROR(FIND("①",A2),0)
   +IFERROR(FIND("②",A2),0)
   +IFERROR(FIND("③",A2),0)=0,
  "該当なし",
  MID(A2,
    IFERROR(FIND("①",A2),0)
   +IFERROR(FIND("②",A2),0)
   +IFERROR(FIND("③",A2),0),1))
```

> **設計意図:** `MID` で固定位置から抽出する方法では、ファイル名に想定外の文字が混入した場合にエラーが発生する。FIND でパターンの存在位置を検索し、見つからなければ 0 を返す方式にすることで、**抽出エラーを最小限に抑えた。** 合計が 0（どのパターンにも該当しない）の場合は「該当なし」を返す。

#### 測定値（`_` と `um` の間）― FIND + MID + 数値変換

```excel
=MID(A2, FIND("_", A2) + 1, FIND("um", A2) - FIND("_", A2) - 1) * 1
```

| ステップ | 処理内容 |
|---------|---------|
| `FIND("_", A2)` | `_` の位置を取得 |
| `FIND("um", A2)` | `um` の位置を取得 |
| `MID(...)` | 2 つの位置の間の文字列を抽出 |
| `* 1` | テキスト → 数値に変換（Spotfire での集計用） |

#### 破断状態（`um` 〜 `.jpg` の間）

```excel
=MID(A2, FIND("um", A2) + 2, FIND(".jpg", A2) - FIND("um", A2) - 2)
```

`um` の直後から `.jpg` の直前までの文字列を抽出。  
破断起点がどこかを判別する**フィルター用の分類データ**として使用。

## キャリアの経緯と転職の動機

### 大学での学び → 技術派遣へ

大学時代はソフトウェアを中心に、**プログラミングや AI の分野を専攻**していました。しかし入社当時は基本情報技術者試験などの実務的な資格を持っておらず、AI もまだ現在ほど注目されていなかったため、ソフトウェア業界は人材が間に合っている状態でした。そのため技術派遣では**製造業、いわゆるハードウェアを中心に**キャリアを積むことになりました。

### 製造現場での苦戦

半導体メモリの製造会社での派遣では、正直に言えば**手作業の精度**で苦戦しました。

- マイクロスコープでの細かなサンプル操作に慣れず、**誤ってサンプルをダメにしてしまう**ことがあった
- データ入力のミスにより、**Spotfire の発表スライドを作り直す**事態が発生した
- 繊細な手作業の連続は、自分にとって得意な領域ではなかった

### 業務効率化での成功体験

一方で、こうした手作業のミスを**仕組みで解決する**取り組みにおいては、まったく違う結果が出ました。

**誰にも頼まれていないにもかかわらず**、自発的にファイル命名規則の設計と Excel 関数・VBA による自動化に取り組み、3〜4 時間の作業を 45 分に短縮。この改善を業務報告で発表したところ、**どの業務報告よりも受けが良かった**という成功体験を得ました。

### プログラマーへの転職を決意した理由

この対比が、自分の適性を明確にしてくれました。

**手を動かす繰り返し作業**では目立つミスがあった一方で、**「どうすればミスが起きない仕組みを作れるか」を考えて実装する**領域では、自発的に動き、成果を出し、高い評価を得ることができた。

大学で学んだプログラミングと AI の知識がこの現場で初めて活き、Excel 関数と VBA という限られたツールでも目に見える改善を生み出せた。ならば、**本格的なプログラミング言語を使えばもっと大きな課題を解決できる**はず。

大学での学びの土台、現場で手を動かして成果を出した実感、そして自分が本当に力を発揮できる領域が明確になったこと――この 3 つが重なったことが、ソフトウェア開発を本業にしたいという決意の原点です。

### この経験から得た技術的な学び

#### ファイル名を「データベース」として設計する発想

ファイル保存時の命名規則という**上流の小さな設計変更**が、下流のすべての工程に効きました。これはソフトウェア設計でいう**データモデリング**の考え方そのものです。

#### 「N 番目の文字を抽出する」だけでは不十分だった

破断方向の判定で、最初は `MID` で固定位置から取得する方法を試しましたが、想定外のケースでエラーを減らすために、`IFERROR + FIND` に切り替えたことで堅牢性を自主的に挙げられた経験は、**エラーハンドリング**の重要性を実務で学ぶ機会になりました。
